{{/* ── What is CaddyUI? ── */}}

What is CaddyUI?

CaddyUI is a web interface for managing Caddy, the modern HTTPS web server. Instead of editing Caddyfile by hand, you click through a UI to manage reverse-proxy rules, redirects, certificates, and more.

Everything you do in the UI is pushed to Caddy's admin API in real time — no restarts, no re-reads.

Who this is for: home-lab users, small teams, anyone with 1 – 20 Caddy servers who wants a faster workflow than editing config files.
{{/* ── First-time setup ── */}}

First-time setup

If you're reading this, you're already logged in — so the hardest part is done. For reference, a full first-time setup looks like this:

  1. Start Caddy with its admin API exposed on :2019 (Docker or bare metal — both work).
  2. Start CaddyUI with CADDY_ADMIN_URL pointed at that address (the sample docker-compose.yml does this automatically).
  3. Visit /setup and create the first admin user. This page only appears when the DB is empty.
  4. Optional: visit Settings and enable Cloudflare Turnstile, 2FA, or email notifications.
Don't expose port 2019 to the internet. It has full control over Caddy. Keep it on localhost, a Docker network, or a WireGuard VPN.
{{/* ── Proxy Hosts ── */}}

Proxy Hosts

A proxy host says: "requests for my.domain.com should be forwarded to this internal service."

Click Proxy Hosts → + New and fill in:

  • Domain names — one or more hostnames (comma-separated).
  • Scheme + Host + Port — where your backend service lives (e.g. http://192.168.1.50:3000).
  • Auto SSL — leave on unless you want to use a custom certificate. Caddy will auto-provision via Let's Encrypt / ZeroSSL.
  • WebSockets / HTTP/2 / HTTP/3 — all on by default.
  • Access control — optional IP allow/deny list.

Hit Save and the rule is live in Caddy immediately. The source domain pill on the list page is clickable — it opens the domain in a new tab to verify.

Tip: use the Enabled toggle to temporarily turn a rule off without deleting it. Great for maintenance windows.
{{/* ── Redirections ── */}}

Redirections

A redirection sends visitors from an old hostname to a new one with a proper HTTP status code. Example: old.example.com → new.example.com.

Pick the right code on the form:

  • 301Moved Permanently — browsers & search engines cache forever. Use for permanent domain moves.
  • 302Found / Temporary — not cached. Use when the redirect may change.
  • 307Temporary Redirect — like 302 but preserves POST/PUT. Safer for APIs.
  • 308Permanent Redirect — like 301 but preserves the HTTP method.

Preserve path keeps the URL path intact: old.com/blog/post → new.com/blog/post. Turn it off to always land visitors on the root.

{{/* ── Advanced routes ── */}}

Advanced Routes

For anything the UI forms can't express — header manipulation, path rewrites, custom matchers — paste raw Caddy JSON here. CaddyUI imports it verbatim into apps.http.servers.srv0.routes.

Syntax check runs on every save. If Caddy rejects the config, you'll see the error inline and the old config stays live.

Heads-up: advanced routes aren't linked to certificates or the host list. You're responsible for their TLS blocks.
{{/* ── Certificates ── */}}

Certificates

For most people, you don't need this page. Caddy auto-provisions certificates via Let's Encrypt (or ZeroSSL as a fallback) whenever you create a proxy host with Auto SSL enabled.

Use the Certificates page only when you need to:

  • Upload a bought / internal CA-issued certificate.
  • Use a DNS-01 challenge (for wildcard certs or hosts not reachable over HTTP).
  • Pin a specific cert to a proxy host instead of letting Caddy manage it.
{{/* ── Cloudflare DNS ── */}}

Cloudflare DNS

If your domains are on Cloudflare, CaddyUI can create and manage the A records for you automatically. This is the recommended setup because Cloudflare's API is fast, well-documented, and free.

  1. First, set the shared Server IP at the top of the Settings DNS section — every provider writes this IP into its A records.
  2. Create a token at dash.cloudflare.com → API Tokens with Zone:Read + DNS:Edit for the zones you want to manage.
  3. Paste the token into Settings → Cloudflare DNS.
  4. Optional: enable Proxied to route traffic through Cloudflare's CDN (orange cloud). Disable to run DNS-only (grey cloud).

When you create a proxy host, pick Cloudflare from the provider dropdown and choose the zone. CaddyUI creates the DNS record, stores its ID, and cleans up on host delete.

If you later change the Server IP, CaddyUI re-points every record it manages across all providers in one background pass. Records it didn't create are never touched.

{{/* ── Porkbun DNS ── */}}

Porkbun DNS

If you registered your domain through Porkbun, CaddyUI can manage DNS A records there too — so your customer never needs to log in to Porkbun to repoint a subdomain.

  1. Go to porkbun.com/account/api and create an API key pair. You'll get a public API Key (starts with pk1_) and a Secret Key (starts with sk1_).
  2. In Settings → Porkbun DNS, paste both keys. The Server IP is shared — set it once at the top of the DNS section and every provider uses it.
  3. This one trips everyone up: for each domain you want CaddyUI to manage, open Porkbun → Domain Management → click Details next to the domain → toggle API Access to ON. The API will reject requests on any domain where that toggle is off, even with valid keys.

When you create a proxy host, pick Porkbun from the provider picker and choose the base domain. CaddyUI will create an A record (or an empty-name record for the apex) pointing at your server IP and remember its ID. Edit → update, delete → cleanup.

Porkbun enforces a minimum TTL of 600 seconds — values below that get silently clamped by the API. Porkbun has no CDN / proxying mode equivalent to Cloudflare's orange cloud.

{{/* ── Namecheap DNS ── */}}

Namecheap DNS

Namecheap's API is the most finicky of the six — worth reading this whole section before enabling it. The rough edges are all Namecheap's, not ours, but CaddyUI shields you from most of them.

  1. Log in to Namecheap and go to Profile → Tools → API Access. Enable API Access and generate a key.
  2. On the same page, add your server's public IP to the Whitelisted IPs list. Every request carries the IP and Namecheap rejects mismatches.
  3. In Settings → Namecheap DNS, fill in: API User (your Namecheap username), API Key, and Whitelisted IP (the same IP you just whitelisted — usually the same as the shared Server IP).

Under the hood: Namecheap's API doesn't support per-record CRUD — every create or delete fetches the full host list, mutates it, and POSTs the whole set back. CaddyUI serialises those calls internally, but if you're hand-editing records in the Namecheap panel at the same time an IP-retarget job is running, you may see one side overwrite the other. Let one finish before starting the other.

Namecheap requires at least 20 active domains on your account or a positive account balance to unlock API access on sandbox-free production. If the API returns "No Permission", that's the gate.

{{/* ── GoDaddy DNS ── */}}

GoDaddy DNS

GoDaddy offers a clean JSON API, but read the tier gate below before wiring it up — it bites a lot of users.

  1. Create a Production key at developer.godaddy.com/keys. You'll get an API Key and an API Secret.
  2. Paste both into Settings → GoDaddy DNS.
Tier gate: as of 2024-04 GoDaddy restricts API access to accounts that hold 10 or more domains or have a Discount Domain Club membership. Keys on smaller accounts return 403 ACCESS_DENIED — that error is surfaced verbatim in the proxy host form so you can see it when it happens. This is not a CaddyUI bug; it's a GoDaddy policy.

Under the hood: GoDaddy doesn't expose stable per-record IDs in its list response. CaddyUI tracks records by the pair type|name — deleting sends DELETE /records/{type}/{name}. If you hand-edit a record's type or name in GoDaddy's panel, CaddyUI may not be able to find it for cleanup on host delete — remove the orphan manually in that case.

{{/* ── DigitalOcean DNS ── */}}

DigitalOcean DNS

DigitalOcean's DNS API is free for any account and Just Works. Recommended if your domain nameservers already point at DigitalOcean (ns1/ns2/ns3.digitalocean.com).

  1. At cloud.digitalocean.com → API → Tokens, create a Personal Access Token with read + write scopes (DNS needs both).
  2. Paste the token into Settings → DigitalOcean DNS.
  3. The domain must already exist in DigitalOcean's Networking → Domains panel. CaddyUI creates records inside existing zones — it doesn't add the zone itself.

Minimum TTL on DigitalOcean is 30 seconds. Nameservers must be pointed at DigitalOcean at the registrar for the record to actually resolve — the API will happily create records on zones whose nameservers point elsewhere, but those records are dark until you repoint.

{{/* ── Hetzner DNS ── */}}

Hetzner DNS

Hetzner's DNS Console is a free, separate product from Hetzner Cloud — you don't need a Hetzner server to use it. Good choice for EU-hosted zones.

  1. Go to dns.hetzner.com → API Tokens and create a token with full DNS permissions.
  2. Paste the token into Settings → Hetzner DNS. The API uses the Auth-API-Token header — CaddyUI sets that automatically, you only paste the token value.
  3. The zone (domain) must already be created in Hetzner DNS Console and the registrar's nameservers must point at hydrogen.ns.hetzner.com + helium.ns.hetzner.de + oxygen.ns.hetzner.com.

Hetzner returns opaque zone IDs (not the domain name) — CaddyUI stores both so the proxy-host list and log lines can display the human-readable zone name.

{{/* ── Import ── */}}

Import from Caddy

Already running Caddy with a hand-written config? You don't have to start from scratch.

  • Import from Caddy — pulls the live JSON config over the admin API and lets you pick which routes to adopt.
  • Paste Caddyfile — dump a Caddyfile in, preview what it will create, then accept.

Imports are non-destructive: CaddyUI creates its own rules alongside yours. Delete the original routes after you confirm everything works.

{{/* ── Snapshots ── */}}

Snapshots & Backup

Every time you save a rule, CaddyUI records a snapshot of the full Caddy config. If something goes wrong, open Snapshots and restore an earlier version with one click.

You can also download the raw config JSON to keep off-box. Admins get a full database backup endpoint for disaster recovery.

{{/* ── Multi-server ── */}}

Multi-server

One CaddyUI can manage many Caddy instances. Add them in Caddy Servers by giving each an Admin URL.

Five ways to connect — pick whatever matches your setup:

  • Same host / Docker — http://caddy:2019 works fine over the internal Docker network.
  • Unix socket (local-only, zero network exposure) — unix:///run/caddy-admin.sock. Set admin unix//run/caddy-admin.sock in your Caddyfile, then share the socket path between containers via a bind-mount. Nothing listens on any TCP port — unreachable from the network by design.
  • WireGuard / Tailscale — wrap traffic in a private overlay and use the peer's tunnel IP. Never expose 2019 on the public internet.
  • Reverse proxy with HTTP Basic Auth — put Caddy (or nginx) in front of port 2019 with basic_auth. Fill in the HTTP Basic Auth section on the server form; CaddyUI sends an Authorization: Basic … header on every call. Simpler than WireGuard when you just need one remote server gated behind a password.
  • SSH reverse-tunnel — ssh -L 2019:localhost:2019 user@host also works for one-off cases.
🚨 Don't ever expose port 2019 directly on the internet. The Caddy admin API has no built-in authentication — whoever reaches it owns your server. Pick one of the options above instead.

The Also deploy to checkboxes on each form let you push the same rule to several servers at once. Certificates are not mirrored — each server provisions its own.

{{/* ── Users & 2FA ── */}}

Users & 2FA

Roles:

  • admin — full access to every page including users, servers, and settings.
  • viewer — read-only. Can see dashboards and lists but can't create / edit / delete.

Each user can enable TOTP 2FA on their profile at /totp/setup. Scan the QR with Google Authenticator, Authy, 1Password, etc., enter the 6-digit code, and it's active on next login.

Admins can require 2FA at signup via Settings and can reset it on any account from Users.

{{/* ── Docker networking ── */}}

Docker networking: why Caddy can't see your container

This trips up everyone running Caddy in Docker. You add a proxy host, the cert issues fine, you open the URL… and get a 502. Caddy's logs say something like:

dial tcp: lookup status-server on 127.0.0.11:53: server misbehaving
status: 502

That error is Docker's internal DNS (127.0.0.11) telling Caddy it has no idea what status-server is. Not an SSL problem, not a Caddy bug — a Docker network problem.

The rule

Docker's embedded DNS only resolves container names on the same user-defined network. If Caddy is on network proxy-net and your upstream container is on status-net (or on the default bridge, which has no DNS at all), Caddy cannot resolve it — full stop.

The fix: one shared network

Create a single network (call it whatever — caddy-net is common) and attach every container Caddy needs to reach. Example docker-compose.yml:

services:
  caddy:
    image: caddy:latest
    networks:
      - caddy-net
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./Caddyfile:/etc/caddy/Caddyfile
      - caddy_data:/data

  caddyui:
    image: applegater/caddyui:latest
    networks:
      - caddy-net
    environment:
      CADDY_ADMIN_URL: http://caddy:2019
    volumes:
      - caddyui_data:/data

  status-server:
    image: louislam/uptime-kuma:1
    networks:
      - caddy-net
    volumes:
      - kuma_data:/app/data

networks:
  caddy-net:
    driver: bridge

volumes:
  caddy_data:
  caddyui_data:
  kuma_data:

Now in CaddyUI's proxy host, the Forward host is simply the service name (status-server) — Docker's DNS resolves it to the container's internal IP.

Already-running containers: bolt them on

If the app you want to expose was already running in a separate docker-compose.yml, you don't have to merge everything. Make the Caddy network external and import it on the other side:

In Caddy's compose file:

networks:
  caddy-net:
    name: caddy-net
    driver: bridge

In the other app's compose file:

services:
  status-server:
    # … existing config …
    networks:
      - default
      - caddy-net

networks:
  caddy-net:
    external: true

The upstream keeps its own private network for talking to its database, AND joins caddy-net so Caddy can reach it.

No recompose? Connect a live container

docker network connect caddy-net status-server

Immediate, no restart. (But write it into compose too — this won't survive docker compose up.)

When the upstream isn't in Docker at all

  • On the same host (a native systemd service, a bare-metal daemon): use host.docker.internal (Docker Desktop) or 172.17.0.1 / host-gateway on Linux.
  • On another machine: use its LAN IP. Make sure the upstream is listening on all interfaces, not just 127.0.0.1.

Quick diagnosis cheat sheet

# what network is Caddy on?
docker inspect caddy --format '{{`{{range $k,$v := .NetworkSettings.Networks}}{{$k}} {{end}}`}}'

# can Caddy resolve the upstream name?
docker exec caddy nslookup status-server
# (or: docker exec caddy wget -qO- http://status-server:3001)

# list everything on a given network
docker network inspect caddy-net

If nslookup works but wget fails, the DNS is fine but the upstream port is wrong — double-check the Forward port in CaddyUI.

Rule of thumb: any new upstream you add needs to either share a network with Caddy or be reached by host/LAN IP. A single caddy-net that every service joins saves a lot of "why can't Caddy see this" debugging later.
{{/* ── FAQ ── */}}

FAQ & Troubleshooting

My domain won't get a certificate. Why?

Check, in order:

  1. DNS A/AAAA record actually points at this server's public IP.
  2. Ports 80 and 443 are open on your firewall and port-forwarded to Caddy.
  3. You're not rate-limited by Let's Encrypt (5 fails per account/hostname/hour).
  4. For wildcards or internal-only hosts, use a DNS-01 challenge — see Certificates.
I changed a rule but the site still shows the old behavior.

Hard-refresh your browser (Ctrl/Cmd + Shift + R). Clients, CDNs, and service workers cache aggressively. On the Caddy side the config changes on save — no restart needed.

My proxy host gets a cert but returns 502 Bad Gateway.

TLS succeeded, but Caddy can't reach the upstream. Usually a Docker network mismatch — if you see lookup name on 127.0.0.11:53: server misbehaving in the logs, your upstream container isn't on the same Docker network as Caddy.

See Docker networking above for the full fix — short version: create one shared caddy-net and attach every container Caddy needs to reach.

"attempt to write a readonly database" / "database is locked"

Readonly: the data volume isn't writable by the container's user (UID 10001). Fix with chown -R 10001:10001 /path/to/caddyui_data and add user: "10001:10001" to your compose file.

Locked: transient — CaddyUI uses WAL + busy-timeout to survive contention. If you see it constantly, your storage is very slow (NFS, SMB, etc.). Put the DB on local disk.

Can I back up just the CaddyUI settings and re-import them?

Yes. Admins can download a full DB backup from /backup. To restore, stop the container, replace /data/caddyui.db with the backup file, start it again.

Is CaddyUI safe to expose on the internet?

Yes, if you put it behind HTTPS (Caddy can reverse-proxy to itself) and enable 2FA + Cloudflare Turnstile in Settings. But the Caddy admin API on port 2019 must never be public — keep it on a private network.

How do I secure port 2019 if I don't have WireGuard or Tailscale?

Caddy's admin API has no built-in password. You have three options that don't need a VPN:

  1. Unix socket (simplest, local-only) — in your Caddyfile's global block, set admin unix//run/caddy-admin.sock. Bind-mount the socket path into the CaddyUI container (e.g. /run/caddy:/run/caddy) and set the server's Admin URL to unix:///run/caddy/caddy-admin.sock. No TCP port is opened at all — nothing to expose.
  2. Reverse proxy with HTTP Basic Auth — put another Caddy block (or nginx) in front of port 2019 with basic_auth. Enter the username/password in the HTTP Basic Auth section when adding the server; CaddyUI will send the credentials on every request. Works across the public internet if you put it behind HTTPS.
  3. UFW / firewall allowlist — limit inbound port 2019 to known source IPs only. Weakest of the three (IP-based auth isn't great), but fine as a belt-and-suspenders layer on top of options 1 or 2. Concrete commands below.

Options 1 and 2 are both supported natively on the Caddy Servers form — you can start with a unix socket for local Caddy and add a remote server behind basic auth without changing anything else.

Example: firewall port 2019 to one source IP only (UFW / iptables)

Use this when CaddyUI and Caddy are on different hosts and you want to block everyone except the CaddyUI host from hitting the admin API. Replace 203.0.113.5 with CaddyUI's public IP (or WireGuard/Tailscale address).

UFW (Ubuntu / Debian):

# On the Caddy host:
sudo ufw allow from 203.0.113.5 to any port 2019 proto tcp
sudo ufw deny 2019/tcp         # everyone else is blocked
sudo ufw reload
sudo ufw status numbered       # verify the rule order — allow must come first

iptables (no UFW):

sudo iptables -A INPUT -p tcp --dport 2019 -s 203.0.113.5 -j ACCEPT
sudo iptables -A INPUT -p tcp --dport 2019 -j DROP
# Persist across reboots:
sudo apt install iptables-persistent
sudo netfilter-persistent save

If Caddy runs in Docker:

Docker bypasses UFW by writing its own iptables rules. Either:

  • Bind Caddy's admin port to the private interface only — in docker-compose.yml use ports: ["10.8.0.1:2019:2019"] instead of "2019:2019" so it never listens on 0.0.0.0, or
  • Don't publish port 2019 at all — attach the CaddyUI container to the same Docker network as Caddy, and set the Admin URL to http://caddy:2019. That's what the shipped docker-compose.yml already does.

Check it worked: from a different machine, run curl -m 5 http://<caddy-host-ip>:2019/config/. It should hang or return "connection refused". From the CaddyUI host it should return JSON.

Example: Caddyfile with Unix socket admin endpoint 
# Caddyfile
{
    admin unix//run/caddy/caddy-admin.sock
}

# …your site blocks…

In docker-compose.yml, share the socket directory between Caddy and CaddyUI:

services:
  caddy:
    volumes:
      - caddy_admin:/run/caddy
  caddyui:
    volumes:
      - caddy_admin:/run/caddy
    environment:
      CADDY_ADMIN_URL: unix:///run/caddy/caddy-admin.sock

volumes:
  caddy_admin:
Example: Caddy admin behind HTTP Basic Auth

Run two Caddy instances (or two site blocks in one instance): one hosts your sites, the other terminates TLS and proxies / to the admin API with basic auth.

admin.example.com {
    basic_auth {
        caddyui $2a$14$…bcrypt-hash…
    }
    reverse_proxy localhost:2019
}

Generate the bcrypt hash with caddy hash-password. Then add the server in CaddyUI with:

  • Admin URL: https://admin.example.com
  • Username: caddyui
  • Password: the plaintext password that matches the hash
